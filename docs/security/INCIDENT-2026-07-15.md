# Repository secret exposure incident — 2026-07-15

## Classification

Production release blocker until containment, credential rotation and history remediation are complete.

## Confirmed exposure

A generated WordPress backup directory was committed to a public repository. The tracked material included non-empty WordPress authentication keys/salts and an installer log containing operational server metadata. Database password fields in the inspected configuration copy were empty; database credential exposure must nevertheless be checked across the complete Git history before closure.

No secret values are reproduced in this document.

## Containment applied in the remediation branch

- Removed the exposed configuration copy from the active tree.
- Removed the generated installer log from the active tree.
- Added explicit ignore rules for WordPress configuration, Duplicator artifacts, backups, database exports, archives, private keys, logs and installer remnants.
- Removed all files that were tracked while also matching `.gitignore`, including WordPress core/runtime/vendor artifacts and generated QA screenshots.
- Added a CI security gate that rejects tracked ignored files, forbidden operational paths and common secret signatures.

## Production actions required for closure

1. Back up the production `wp-config.php` outside the public web root with restrictive permissions.
2. Replace all WordPress authentication keys and salts with cryptographically random values.
3. Validate PHP syntax and WordPress bootstrap after the atomic replacement.
4. Invalidate all administrator sessions.
5. Move Duplicator backup and installer remnants outside the public web root.
6. Review the administrator account list and investigate unrecognized accounts.
7. Rotate the database password if any historical object contains a non-empty value, or if the hosting backup/archive exposure cannot be ruled out.
8. Temporarily make the repository private while history remediation and GitHub cache cleanup are coordinated.

## Git history remediation

Deleting files in a normal commit does not remove prior blobs. The complete repository history, branches and tags must be rewritten with `git filter-repo` after production secrets are rotated. All collaborators must discard old clones and re-clone after the force push. GitHub Support may need to remove cached views, pull-request refs or unreachable sensitive objects.

Minimum paths to purge:

```text
wp-content/backups-dup-lite/
wp-content/backups-*/
**/source_site_wpconfig
**/dup-installer/**
**/*installer-log*
**/entries_stored.json
```

The rewrite must also remove historical WordPress core, uploads, third-party plugin copies, generated logs, archives, database exports and QA screenshots that are excluded by the canonical `.gitignore`.

## Release gate

Production deployment remains `NO GO` until:

- production keys/salts are rotated;
- exposed server artifacts are outside the web root;
- repository history is rewritten and verified;
- the repository security workflow passes;
- the current administrator and remote-editing plugin inventory is approved;
- the repository is private or a documented decision explicitly accepts public visibility after all sensitive history is removed.
